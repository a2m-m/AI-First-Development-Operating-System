# Codex コンテキスト受け渡しガイド

Claude が Codex にコンテキストを渡す際の品質基準と設計指針。
`/review`・`/research`・`/issue-lint` の比較調査から導出したチェックリスト。

---

## 背景

Claude→Codex のコンテキスト受け渡し品質が、コラボレーション結果（レビュー精度・リサーチ深度・評価安定性）を大きく左右する。
コンテキストが貧しいと Codex は方向感を失い、質問が増え、出力が不安定になる。

---

## 良いコンテキストの 7 要素

### 1. タイトルで目的を一言で伝える

**悪い例**: コンテキストの冒頭が本文から始まる
**良い例**: `# レビュー対象: PR #108 — コンテキスト受け渡し品質評価`

Codex のタスクが見出し一行で分かると、初回応答の的中率が上がる。

---

### 2. 役割（ロール）を明示する

**悪い例**: タスクだけを渡して Codex に役割を推測させる
**良い例**: `あなたは Issue レビュアーです。以下の Issue を読み、品質を評価してください。`

ロール定義は Codex の応答一貫性を高める。`/issue-lint` がこれを実践しており、E2E で最も安定した動作を示した。

```markdown
# <タスク種別>依頼

あなたは <役割> です。<一文で何をするかを説明する>。
```

---

### 3. 評価・調査観点を列挙する

**悪い例**: 「レビューしてください」だけを渡す
**良い例**:
```markdown
## 評価観点

1. **AC の明確さ**: 各 AC が Yes/No で判定できるか
2. **Commit Plan の妥当性**: 1〜3コミットに適切に分割されているか
3. **スコープの適切さ**: Out of Scope が明示されているか
```

観点を列挙することで Codex が「何を見ればいいか」を把握した状態で開始できる。質問が減り、出力が構造化される。

**観点数の目安**: 3〜5 項目。多すぎると焦点が散る。

---

### 4. 情報の空白を事前に埋める

**悪い例**: 「このテーマを調べてください」だけを渡す
**良い例**: `/research` の二段階フロー — Explore エージェントが事前調査し、その結果を含めて Codex に渡す

```markdown
## 事前調査結果

### 調査スコープ
<Explore が特定したコードベース上の対象領域>

### 対象ファイル一覧
<関連ファイルとその役割>

### 現状サマリ
<わかっていること・わかっていないこと>
```

「調べてください」ではなく「調べた結果はこれです、ここから深掘りを」と渡すことで Codex の探索範囲が絞られる。

---

### 5. 期待する出力形式を示す

**悪い例**: 出力形式の指定なし → Codex が好みで整形する → 呼び出し元が整形に苦労する
**良い例**:
```markdown
## リサーチ指示

最終的に「## リサーチレポート」として以下の形式で報告をまとめてください：

- ## 調査観点と結果
- ## 発見事項
- ## 未解決の問い
- ## 結論
```

出力形式を示すと、後続の整形・投稿処理がシンプルになる。

---

### 6. AC・Non-goals を含める

**悪い例**: PR の差分だけを渡す
**良い例**: `/review` の PR フロー — PR 差分 + Issue の AC + Non-goals を合わせて渡す

```markdown
## 関連 Issue の Acceptance Criteria
<Issue の ## Acceptance Criteria セクション>

## Non-goals
<Issue の ## Out of Scope セクション>

## 差分
<gh pr diff の全出力>
```

「何を満たすべきか」と「何を無視すべきか」の境界を示すことで、スコープ外の指摘（ノイズ）が減る。

---

### 7. セクション構造（H2/H3）で情報を分割する

**悪い例**: 長い平文テキスト
**良い例**: H2/H3 の見出しでセクション分け

全スキル共通のベストプラクティス。構造化テキストは Codex のパース精度が高く、セクション単位で参照しやすい。セクション名は「何の情報か」を明示する名詞句にする。

## スキル別コンテキスト評価

| 要素 | `/review`（PR） | `/review`（Issue） | `/research` | `/issue-lint` |
|---|---|---|---|---|
| 1. タイトル | ✅ | ✅ | ✅ | ✅ |
| 2. ロール定義 | ❌ | ❌ | ❌ | ✅ |
| 3. 観点列挙 | ✅（AC/Non-goals） | ⚠️ | ✅（Explore 観点） | ✅（5項目） |
| 4. 事前調査 | ✅（差分・AC） | ⚠️（関連コード任意） | ✅（Explore 二段階） | ❌ |
| 5. 出力形式 | ✅（Severity 区分） | ✅（Severity 区分） | ✅（レポート形式） | ❌ |
| 6. AC/Non-goals | ✅ | ❌ | N/A | N/A |
| 7. 構造化 | ✅ | ✅ | ✅ | ✅ |

> ✅ 実践済み / ⚠️ 部分的 / ❌ 欠落

## 新スキルを作る際のコンテキストテンプレート

```markdown
# <タスク種別>依頼: <対象の一言説明>

あなたは <役割> です。<何をするかを一文で説明>。

## <役割>の観点

1. **<観点1>**: <判定基準>
2. **<観点2>**: <判定基準>
3. **<観点3>**: <判定基準>

## 事前調査 / 関連情報

<Explore または gh コマンドで取得した情報>

## <対象> 内容

<gh issue view / gh pr diff / 対象コードなど>

## 期待する出力形式

<出力フォーマットの見本またはセクション名一覧>
```

## 既知の問題と改善候補

以下は本調査で発見した改善候補。実装変更は別 Issue で対応する。

| # | 対象スキル | 問題 | 影響 | 追跡 |
|---|---|---|---|---|
| 1 | `/review`（Issue フロー） | 「関連コード」の収集方法が不定 | コンテキストが薄くなることがある | 未起票 |
| 2 | `/review`・`/research` | ロール定義が欠落 | 応答の一貫性がやや低い | #121 |
| 3 | `/issue-lint` | 期待出力フォーマットがコンテキストに含まれない | 整形の一貫性が低下することがある | #122 |
| 4 | 全スキル | `reply` がログ全文を再送しコンテキストが肥大化 | 後続ターンの品質低下 | #115 |
