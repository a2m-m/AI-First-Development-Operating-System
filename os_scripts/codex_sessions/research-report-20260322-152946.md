# リサーチレポート: Claude Code のスキルシステム設計パターン

## 調査観点と結果

### 1. `codex_session.sh` の実装方式
- `start` は `session_id` と `context_file` を検証し、`os_scripts/codex_sessions/<id>.log` を Markdown で初期化したうえで、必要なら `AGENTS.md` を先頭注入して `codex exec -` を1回実行する
- `reply` は永続セッションに接続せず、既存ログ全文を `cat` して新しい Claude メッセージを足し、再度 `codex exec -` を実行する（stateless な「ログ再構成型」）
- `end` は終了日時をログ末尾に追記するだけで、Codex 側のセッション破棄や後始末はない
- `run_codex()` は `stderr` も含めて丸ごと取り込む

### 2. 14スキル分類表
| スキル | 分類 | 主責務 | 主な受け渡し先 |
|---|---|---|---|
| `context-sync` | 状態同期 | `.ai-context.md` と open issues 読み込み | 作業文脈 |
| `istart` | 着手選定 | 次 Issue 選定、branch 作成、`/plan` への橋渡し | branch / plan mode |
| `issue-create` | 命令書生成 | テンプレ準拠 Issue 作成、任意で Codex 品質評価 | Issue / comment |
| `issue-lint` | 命令書審査 | ルールベース + Codex で Issue 品質検査 | Issue comment |
| `plan` | 実装準備 | AC / Non-goals / Commit Plan を実装計画化 | 実装計画 |
| `review` | 品質レビュー | PR/Issue を Codex 主導で審査 | PR/Issue comment |
| `research` | 調査 | Explore でスコープ定義し Codex に深掘りさせる | research report |
| `gate` | 品質ゲート | `./os_scripts/run ci` 実行と結果解釈 | PASS/FAIL |
| `commit-lint` | 補助ゲート | コミット規約検査 | lint report |
| `pr-complete` | 事後同期 | PR マージ後の `.ai-context.md` 更新 | 次セッション状態 |
| `ci-failure-triage` | 失敗吸収 | CI 失敗から bug Issue 化 | Bug Issue |
| `release-notes` | 出荷補助 | タグ間変更をリリースノート化 | note file/stdout |
| `skill-create` | メタスキル | 新スキルの scaffold と frontmatter 設計 | `SKILL.md` |
| `project-setup` | ブートストラップ | 複製後の初期化・資材点検・公開設定 | 初期化済み repo |

### 3. コンテキスト受け渡し図
```
Issue template
  -> issue-create
  -> issue-lint
  -> plan
  -> 実装
  -> review
  -> gate
  -> PR merge
  -> pr-complete
  -> .ai-context.md
  -> context-sync / istart
```
```
調査系:
research
  -> Explore で対象定義
  -> CONTEXT_FILE
  -> codex_session start/reply/end
  -> report + session log
```
- 受け渡しは関数呼び出しではなく artifact 連鎖
- 意図と仕様は Issue/PR、運用状態は `.ai-context.md`、Codex 対話は `CONTEXT_FILE` と `*.log` に落ちる
- フォールバックは非対称: `review`/`research` は Codex 基盤がなければ停止、`issue-lint`/`issue-create --review` はルールベースのみで劣化運転

## 発見事項（設計パターン）

### べき・べからず一覧

**べき：**
- Issue を命令書として扱い、AC を Yes/No、Non-goals、Commit Plan まで明示する
- `1スキル = 1責務` を守り、複数目的を混ぜない
- `allowed-tools` は最小権限で切る
- 開始前に `.ai-context.md` を読み、終了後に更新する
- Template のスキルは Instance で改造せず、改善は Template に還元する

**べからず：**
- AC が曖昧なまま実装を始める
- `/review` と `/gate` を飛ばして完了扱いにする
- スキル間で暗黙メモリに依存する（受け渡しは artifact に落とす）
- `.ai-context.md` を更新せずにセッションを終える
- 単発作業や `os_scripts/run` で済む単純実行まで無理にスキル化する

## 設計上の不整合リスト

1. **HIGH** `/plan` と `/gate` は `CLAUDE.md` では Claude 自動起動だが、frontmatter では `disable-model-invocation: true`（ユーザーのみ起動）と矛盾
2. **HIGH** `context-sync` は `.ai-context.md` 更新責務を持つが、`allowed-tools` に `Edit`/`Write` がなく実行不可
3. **MEDIUM** Template Repo ルール（`.ai-context.md` に個別案件の Issue/PR を書かない）と実ファイルの状態がズレている
4. **MEDIUM** `--no-inject` オプションは実装済みだが、呼び出し側スキルが全て通常 `start` を使用。また `reply` のログ全文再送が CLI ヘッダ・MCP エラーを次ターンに持ち越す
5. **MEDIUM** `review`/`research`/`issue-lint`/`issue-create --review` は「途中エラーでも `end` を必ず実行」と書くが、`trap`/finally 相当の保証がない
6. **MEDIUM** `context: fork`/`agent: Explore|Plan` は frontmatter リファレンスにあるが現行スキルでは未使用（実運用は `allowed-tools: Agent(subagent_type=Explore)`）
7. **LOW** `issue-create --review` のセッション ID 接頭辞が `issue-lint-` になっており役割名とログ名が不一致

## 未解決の問い
- `context: fork`/`agent:` メタデータが将来拡張用なのか廃止予定なのか不明

## 結論
スキルシステムは「artifact-driven な設計」を採用しており、スキル間の受け渡しは暗黙メモリではなく Issue/PR・`.ai-context.md`・一時ファイル・セッションログの4系統に落とす方式。Codex 連携は「ログ再構成型」の stateless セッションで実現している。HIGH 不整合として `plan`/`gate` の frontmatter 矛盾と `context-sync` の権限不足を早急に解消すべき。
